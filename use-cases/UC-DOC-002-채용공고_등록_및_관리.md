# UC-DOC-002 채용공고 등록 및 관리

| 항목 | 내용 |
|------|------|
| **액터** | 인증 사용자 (`AuthenticatedUser`) |
| **목표** | 지원하려는 채용공고(Job Description)를 URL로 등록하고 관리한다 |
| **사전 조건** | 로그인 상태 |
| **사후 조건** | 등록된 채용공고가 벡터 DB에 저장되어 면접 질문 생성 시 맥락 자료로 활용 가능한 상태가 된다 |

---

## 기본 흐름

1. 사용자가 채용공고 관리 페이지에 접근한다
2. 사용자가 채용공고 URL을 입력한다
   - 필요 시 식별용 제목(`custom_title`)을 직접 입력할 수 있다 (미입력 시 크롤링된 공고 제목 사용)
3. 시스템이 해당 URL이 이미 `job_descriptions`에 존재하는지 확인한다
   - **존재하는 경우**: 크롤링 없이 기존 원문을 재사용하고 `user_job_descriptions`에 북마크만 추가한다
   - **존재하지 않는 경우**: `job_descriptions`에 새 행을 생성하고 크롤링을 비동기로 시작한다
4. 시스템이 즉시 응답을 반환하고, 백그라운드에서 크롤링 → LLM 파싱 → 벡터 DB 저장을 처리한다
5. 채용공고가 사용자 목록에 추가되고 기본적으로 활성화 상태로 설정된다

---

## 대안 흐름

- **2a.** 유효하지 않은 URL 형식인 경우 오류 메시지를 표시하고 재입력을 유도한다
- **3a.** 동일 사용자가 이미 북마크한 URL인 경우 중복 등록 안내 메시지를 표시한다
- **4a.** 크롤링에 실패한 경우 UI에 "공고 정보를 불러오지 못했습니다" 상태를 표시한다
- **5a.** 사용자가 기존 채용공고를 비활성화하면 면접 질문 생성에서 제외된다
- **5b.** 사용자가 채용공고를 삭제하면 `user_job_descriptions` 행이 제거된다 (공유 원문 `job_descriptions`는 유지)

---

## 비고

- 채용공고 원문(`job_descriptions`)은 URL 기준으로 공유된다. 동일 URL을 여러 사용자가 등록해도 크롤링은 최초 1회만 수행된다
- 사용자 레이어(`user_job_descriptions`)는 각 사용자의 북마크 및 활성화 상태를 독립적으로 관리한다
- 크롤링·파싱은 비동기 처리: 등록 즉시 응답 반환 후 백그라운드에서 처리되며, 완료 전까지 UI에 "분석 중..." 상태를 표시한다
- 채용공고는 지원 현황(`Application Context`)과 별개로 독립적으로 관리된다
- 등록된 채용공고는 [UC-DOC-003-지원_현황_등록_및_관리.md](./UC-DOC-003-지원_현황_등록_및_관리.md)에서 지원 현황에 연결할 수 있다
- 관련 테이블: `job_descriptions`, `user_job_descriptions`
- 벡터 DB: `user_document_chunks` → [vector-db-schema.md](../vector-db-schema.md)
