# 매일메일 가상 면접 버전 — 기술 조사 보고서

> 작성일: 2026-03-07 | 버전: 2.0

## 1. 프로젝트 개요

"매일메일 가상 면접 버전"은 사용자가 매일 설정한 개수만큼 AI 기반 모의 면접 질문을 받고, 텍스트(MVP) 또는 영상/음성(V2)으로 답변하며, AI가 답변의 적합성을 평가·피드백하는 서비스다. 이력서, 기술 스택, 목표 회사 등을 기반으로 개인화된 면접 질문을 생성하며, 습관 형성(streak) 메커니즘을 통해 매일 꾸준한 면접 연습을 유도한다.

## 2. LLM 모델 선정 및 전략

### 2.1 상용 API 모델 비교 (2026 기준)

면접 질문 생성 플랫폼에서 LLM 선택의 핵심 기준은 **한국어 품질**, **요청당 비용**, **구조화된 출력(Structured Output) 신뢰성**이다.

**GPT-4.1 mini** (OpenAI): 입력 $0.40 / 출력 $1.60 (백만 토큰당), 100만 토큰 컨텍스트 윈도우. 한국어 의료면허시험에서 83.2% 정확도로 전체 모델 중 최고 성능을 기록했다. 코드스위칭(한영 혼용) 최소화 측면에서 가장 안정적이며, 질문 생성에 최적이다.

**GPT-5 mini** (OpenAI): 입력 $0.25 / 출력 $2.00. GPT-4.1 mini 대비 입력 비용이 저렴하여, 긴 이력서를 컨텍스트로 넣는 질문 생성 시나리오에 유리하다.

**Claude Sonnet 4.6** (Anthropic): 입력 $3 / 출력 $15. 가격이 높지만, 뉘앙스 있는 평가와 구조화된 피드백에서 강점이 뚜렷하다. 루브릭 기반 답변 평가(LLM-as-Judge)에 적합하다.

**Gemini 2.5 Flash** (Google): 입력 $0.15 / 출력 $3.50. 가장 저렴한 옵션이며 무료 티어가 넉넉하여 프로토타이핑에 유리하지만, 한국어 성능은 GPT-4o 계열 대비 다소 부족하다.

**오픈소스 모델 중 한국어 최강: Qwen 3** (Alibaba): CJK 언어(한중일), 관용어, 기술 용어 처리에 탁월하다. SiliconFlow 호스팅 기준 입력 $0.35 / 출력 $1.42로 상용 수준 품질을 오픈소스 가격에 제공한다.

**한국어 네이티브 모델**: Naver HyperCLOVA X(비공개 API), LG EXAONE-3(오픈소스). 데이터 주권이 중요한 B2B/공공 사업에서 고려할 수 있다.

### 2.2 권장 전략: 용도별 모델 분리

| 용도 | 권장 모델 | 예상 비용/건 |
|------|----------|-------------|
| 질문 생성 | GPT-4.1 mini 또는 GPT-5 mini | ~₩2 (≈$0.0015) |
| 상세 답변 평가 | GPT-4.1 또는 Claude Sonnet 4.6 | ~₩13~65 (≈$0.01~0.05) |
| 비긴급 배치 작업 | OpenAI Batch API (50% 할인) | 위 가격의 절반 |

10,000명 활성 사용자가 월 50,000건의 평가를 생성할 경우, 총 LLM 비용은 **월 $625~$2,850** 수준이다.

## 3. LLM 통합 아키텍처: LangChain 대안 탐색

### 3.1 왜 LangChain을 피하는가

LangChain은 프로토타이핑에 강력하지만, 프로덕션 환경에서 여러 문제가 보고되고 있다. 주간 단위로 API가 변경되어 기존 코드가 빈번히 깨지며, 다중 추상화 레이어가 디버깅을 어렵게 만든다. 면접 질문 생성 → 평가처럼 **명확한 프롬프트-응답 패턴**에서는 과도한 프레임워크다.

### 3.2 권장 대안: Instructor + LiteLLM 조합

**Instructor** (월 300만+ 다운로드, GitHub 11K+ stars): Pydantic 기반의 구조화된 출력 라이브러리. LLM 응답을 Pydantic 모델로 자동 검증하고, 실패 시 자동 재시도한다. 15개 이상의 LLM 프로바이더(OpenAI, Anthropic, Google, Ollama 등)를 단일 인터페이스로 지원한다.

```python
import instructor
from pydantic import BaseModel

class InterviewFeedback(BaseModel):
    overall_score: int           # 1-10
    technical_accuracy: int      # 1-10
    structure_score: int         # 1-10
    strengths: list[str]
    improvements: list[str]
    model_answer: str

client = instructor.from_provider("openai/gpt-4.1-mini")
feedback = client.chat.completions.create(
    response_model=InterviewFeedback,
    messages=[{"role": "user", "content": prompt}],
    max_retries=3,
)
```

이 방식의 장점은 명확하다. JSON 파싱, 에러 핸들링, 재시도를 프레임워크가 처리하며, Pydantic 검증 실패 시 자동 재시도로 출력 품질을 보장한다. 타입 안전성과 IDE 지원이 완벽하다.

**LiteLLM** (멀티 프로바이더 게이트웨이): 100개 이상의 LLM을 OpenAI 호환 인터페이스로 통합한다. Netflix 등 엔터프라이즈에서 사용 중이며, 프로바이더 간 전환이 코드 한 줄로 가능하다. Instructor와 조합하면 프로바이더 독립적인 구조화된 출력 파이프라인을 구축할 수 있다.

**PydanticAI** (Pydantic 팀 공식 에이전트 런타임): Instructor의 상위 호환으로, 에이전트 워크플로우, 내장 옵저버빌리티, 리플레이 가능한 데이터셋을 제공한다. V2에서 AI 면접관 아바타(꼬리질문 생성)를 구현할 때 적합하다.

### 3.3 통합 아키텍처 권장안

| 계층 | 도구 | 역할 |
|------|------|------|
| 구조화된 출력 | **Instructor** | 질문 생성, 평가 결과의 Pydantic 모델 자동 검증 |
| 멀티 프로바이더 라우팅 | **LiteLLM** | GPT/Claude/Gemini 간 전환, 페일오버, 비용 추적 |
| 에이전트 워크플로우 (V2) | **PydanticAI** | 꼬리질문 생성, 대화형 면접 시뮬레이션 |
| 옵저버빌리티 | **Langfuse** (오픈소스) | 프롬프트 관리, 비용 추적, 트레이싱 |

## 4. 답변 평가(Evaluation) 파이프라인

### 4.1 3단계 하이브리드 평가 아키텍처

**Tier 1 — 빠른 스크리닝 (임베딩 기반 유사도)**: OpenAI text-embedding-3-small 또는 Sentence-Transformers로 사용자 답변과 모범 답변의 벡터 유사도를 측정한다. 건당 약 ₩1.3($0.001)로, 관련성 빠른 점검에 사용한다.

**Tier 2 — 상세 평가 (LLM-as-Judge)**: Zheng et al.(NeurIPS 2023)이 제안한 LLM-as-Judge 방식은 GPT-4 심사위원이 인간 선호도와 80% 이상 일치한다는 것을 입증했다. G-Eval 프레임워크는 Chain-of-Thought 추론을 구조화된 루브릭 평가에 추가하여, 인간 판단과 Spearman 상관계수 0.514를 달성했다.

면접 답변 평가는 4~6개 차원으로 분해해야 한다: 기술적 정확성, 완전성, 명확성, 구조(STAR 기법), 관련성, 예시 활용. **모범 답변을 함께 제공하면 실패율이 70%에서 15%로 감소**한다. 알려진 바이어스로는 장문 편향(LLM이 긴 답변을 선호)과 자기 강화 편향(GPT-4가 자체 출력을 ~10% 우대)이 있다.

**Tier 3 — 오픈소스 대안 (Prometheus 2)**: Mixtral 8x7B 기반 오픈소스 평가 모델로, 인간 판단과 72~85% 일치도 및 Pearson 상관계수 0.897을 달성한다. 온프레미스 배포로 데이터 프라이버시를 보장하면서 질문 유형별 커스텀 루브릭을 적용할 수 있다.

### 4.2 피드백 출력 구조 (Pydantic 모델)

```python
class InterviewEvaluation(BaseModel):
    overall_score: int                    # 1-10
    dimensions: dict[str, DimensionScore] # 차원별 점수 + 피드백
    strengths: list[str]                  # 구체적 강점
    improvements: list[str]              # 개선 제안 (행동 가능한)
    model_answer_highlights: list[str]   # 모범 답변 핵심 포인트
    star_analysis: STARAnalysis          # STAR 기법 분석 (선택)
```

## 5. 이력서 파싱 파이프라인

### 5.1 문서 추출 + LLM 구조화 추출 조합

**PDF 추출**: PyMuPDF(텍스트), python-docx(DOCX). 스캔 문서는 Naver CLOVA OCR(한국어 최적) 또는 Tesseract OCR(한국어 팩 포함) 사용.

**LLM 기반 엔티티 추출**: GPT-4o-mini의 구조화된 출력(Instructor + Pydantic 스키마)으로 이름, 기술 스택, 경력, 학력, 자격증을 추출한다. 건당 약 ₩1.3($0.001). arXiv 연구(2510.09722)에 따르면, 섹션별 분리 추출(기본 정보, 경력, 학력, 기술)이 단일 패스 추출보다 현저히 정확하다.

### 5.2 한국어 이력서 특수 고려사항

한국 이력서 고유의 과제: 표(table) 중심 포맷, 한영 혼용 텍스트, 날짜 형식 차이(YYYY.MM vs YYYY년 MM월). **한영 이중 기술 분류 체계**(taxonomy)를 유지하여 "스프링 부트" ↔ "Spring Boot", "자바" ↔ "Java" 등의 매핑을 정확히 처리해야 한다. 전용 Resume Parser API(Affinda: 건당 $0.02~0.13, 56개 언어 지원)도 존재하지만, LLM 기반 파싱이 비용·유연성에서 우위다.

## 6. 음성·영상·보이스 분석 (V2 기술 스택)

### 6.1 음성-텍스트 변환 (STT)

**Naver CLOVA Speech** (~분당 $0.003): 네이버의 방대한 한국어 데이터 코퍼스로 훈련되어 한국어 인식 정확도 최고. 한국어 면접 답변 STT의 1차 선택지다.

**Deepgram Nova-3** (배치 $0.0043/분, 스트리밍 $0.0077/분): 한국어 최적화(WER 27% 감소), 300ms 이하 스트리밍 레이턴시. 영어 면접이나 멀티링구얼 시나리오에 강하다.

**OpenAI Whisper API** ($0.006/분): 99개 언어 지원, 안정적 정확도. 셀프 호스팅은 **faster-whisper**(CTranslate2 구현)로 바닐라 Whisper 대비 4배 빠른 처리(INT8 양자화).

**권장**: CLOVA Speech(한국어 1차) + Deepgram Nova-3(영어/백업) 이중 프로바이더 전략.

### 6.2 표정·바디랭귀지 분석

**MediaPipe Face Mesh** (Google, Apache 2.0): 468개 3D 얼굴 랜드마크 실시간 추적, 모바일에서도 동작. 시선 추적(홍채 랜드마크), 고개 자세 추정, 자세 분석(MediaPipe Pose 33개 신체 키포인트) 가능.

감정 탐지는 AffectNet 훈련 Vision Transformer 또는 DeepFace 라이브러리(8개 감정 카테고리, ~60-65% 정확도)를 MediaPipe 얼굴 ROI 위에 레이어링한다.

**Hume AI Expression Measurement API** ($0.0828/분 영상+음성): 상용 멀티모달 분석 중 가장 풍부한 감정 분석. 프리미엄 티어용으로 고려한다.

**윤리적 고려사항**: Microsoft Azure는 신뢰성 우려로 감정 인식을 폐지했고, 한국 개인정보보호법(PIPA)은 생체 데이터에 엄격한 조항을 두고 있다. 모든 얼굴 분석은 자기 개선 코칭으로 프레이밍하고, 명시적 동의 플로우를 구현하며, 원본 영상 보존을 최소화해야 한다.

### 6.3 음성 운율(Prosody) 분석

**Praat/parselmouth**: 음성학 분석 표준 도구(피치 추적, 포먼트, 지터, 시머). **librosa**: 발화 속도, 스펙트럼 특성, 강도 윤곽. **myprosody**: 발화 능숙도 측정 전용(발화 속도, 필러 워드, TOEFL 스케일 점수).

필러 워드 감지: 트랜스크립트 키워드 매칭("음", "어", "그", "um", "uh") + 오디오 특성 교차검증 하이브리드. **목표 벤치마크**: 영어 130~160 WPM, 한국어 250~350 음절/분; 필러 워드 비율 분당 3회 이하.

### 6.4 아키텍처: 비동기 우선, 실시간은 나중에

배치 처리 파이프라인(녹화 → S3 업로드 → 비동기 처리 → 결과 대시보드)이 가치의 90%를 비용의 20%로 전달한다. 10분 면접 처리 비용: **비동기 $0.08~0.90** vs 실시간 GPU 상시 가동 $0.20+. 비동기 모드에서는 영상 분석을 30fps 대신 1~5fps로 처리할 수 있어 6~30배 컴퓨팅 절감. 실시간 코칭(라이브 필러 카운트, 페이스 인디케이터, 아이컨택 상태)은 V2 프리미엄 기능으로, WebRTC + LiveKit 미디어 오케스트레이션을 사용한다.

## 7. Task Queue: Celery 대안 분석 및 AWS 네이티브 전략

### 7.1 Celery의 한계

Celery는 Python 비동기 태스크 큐의 사실상 표준이지만, 프로덕션에서 여러 문제가 보고된다. 수동 스케일링 필요, 서버 직접 관리(Redis/RabbitMQ 브로커), SQS 백엔드 사용 시 리소스 누수와 visibility timeout 제약(최대 12시간), 대규모에서의 모니터링·디버깅 어려움 등이다.

### 7.2 권장안: AWS SQS + Lambda + EventBridge Scheduler

본 프로젝트의 태스크는 크게 두 가지다: (1) **스케줄링 태스크** — 매일 정해진 시간에 사용자별 면접 질문 생성·발송, (2) **비동기 처리 태스크** — 답변 평가, 영상 분석 등 무거운 AI 작업. AWS 네이티브 서비스로 양쪽 모두 커버할 수 있다.

**Amazon EventBridge Scheduler** (스케줄링):
- 서버리스 스케줄러로 cron, rate, 일회성 스케줄 지원
- 무료 티어: 월 1,400만 호출 무료
- 200개 이상 AWS 서비스를 직접 타겟으로 호출
- 타임존 지원(한국 KST 직접 설정 가능), 여름시간 자동 처리
- 재시도 정책, DLQ(SQS) 설정 가능
- 매일 아침 7~9시 사용자별 질문 발송을 스케줄링하는 데 최적

**Amazon SQS** (비동기 태스크 큐):
- 완전 관리형, 무제한 확장, 인프라 관리 불필요
- FIFO 큐로 메시지 순서 보장 가능
- AWS Lambda, ECS Fargate 등과 네이티브 통합
- 메시지 보존 최대 14일
- 종량제 과금 (소~중규모에서 매우 경제적)

**AWS Lambda** (컴퓨팅):
- SQS 트리거로 자동 스케일링
- 요청 기반 과금, 유휴 비용 제로
- 최대 15분 실행 시간, 10GB 메모리
- LLM API 호출(질문 생성, 평가) 처리에 적합 (보통 수초~수십초)

**AWS Step Functions** (복잡한 워크플로우):
- 영상 분석처럼 다단계 처리(STT → 음성 분석 → 표정 분석 → 종합 평가)에 적합
- 시각적 워크플로우 에디터, 병렬 실행, 에러 핸들링
- EventBridge Scheduler와 직접 연동 가능

### 7.3 태스크별 아키텍처 매핑

| 태스크 | AWS 서비스 조합 | 설명 |
|--------|----------------|------|
| 매일 질문 발송 | EventBridge Scheduler → Lambda → SES/알림톡 | cron(0 7-9 * * ? *) KST |
| 답변 평가 (텍스트) | API Gateway → SQS → Lambda | 10초 이내 완료 |
| 영상/음성 분석 (V2) | S3 업로드 → SQS → Step Functions → Lambda(s) | 다단계 파이프라인 |
| streak 알림 | EventBridge Scheduler → Lambda → SQS → 알림톡/SES | 매일 저녁 미완료자 푸시 |
| 주간 리포트 생성 | EventBridge Scheduler → Step Functions | 매주 월요일 집계 |

### 7.4 Celery vs AWS 네이티브 비교

| 기준 | Celery + Redis/SQS | AWS 네이티브 (SQS + Lambda + EventBridge) |
|------|--------------------|-----------------------------------------|
| 인프라 관리 | 워커 서버·브로커 직접 관리 | 완전 서버리스, 관리 불필요 |
| 스케일링 | 수동 워커 수 조정 | 자동 스케일링 |
| 스케줄링 | Celery Beat (서버 필요) | EventBridge Scheduler (서버리스) |
| 비용 (100 사용자) | EC2 t3.small ~$15/월 + Redis | Lambda 무료 티어 내 |
| 비용 (10,000 사용자) | EC2 2~3대 ~$50~100/월 | Lambda ~$10~30/월 + SQS ~$5/월 |
| 모니터링 | Flower 별도 설정 | CloudWatch 네이티브 통합 |
| 장애 복구 | 수동 대응 필요 | DLQ, 재시도 정책 내장 |

## 8. 알림 시스템: 카카오톡 알림톡 + 이메일 이중 채널

### 8.1 이중 채널 전략

한국 시장에서 이메일만으로는 충분한 도달률을 확보하기 어렵다. 카카오톡의 거의 보편적인 보급률(스마트폰 사용자 95%+)을 활용하여, **알림톡을 1차 채널, 이메일을 2차 채널(보조 및 상세 콘텐츠 전달)**로 운영한다.

### 8.2 카카오톡 알림톡

카카오 비즈니스 채널을 통해 발송하는 정보성 메시지로, 사전 승인된 템플릿 기반이다. 사용자가 카카오톡 채널을 추가하지 않아도 수신 가능하다는 점이 친구톡 대비 강점이다.

- **발송 비용**: 건당 약 7~10원
- **도달률**: 90%+ (이메일 대비 3~5배)
- **오픈률**: 70~80% (이메일 ~35% 대비 2배+)
- **연동 방식**: 카카오 i 커넥트톡 API 또는 NHN Cloud, 비즈뿌리오 등 대행사 API
- **템플릿 예시**: "오늘의 면접 질문이 도착했습니다 🎯\n[#{질문번호}] {질문 미리보기}\n\n▶ 답변하러 가기: {URL}\n연속 {streak_count}일째 도전 중!"

### 8.3 AWS SES (이메일)

- **비용**: 1,000통당 $0.10, EC2에서 월 62,000통 무료
- **역할**: 상세 피드백 리포트, 주간 요약, 계정 관련 알림
- **모범 답변, 차원별 점수, 개선 제안** 등 긴 콘텐츠는 이메일이 적합
- **Bounce/Complaint 관리**: SES의 SNS 연동으로 자동 처리

### 8.4 알림 발송 아키텍처

```
EventBridge Scheduler (매일 07:00 KST)
  → Lambda (사용자별 질문 생성)
    → SQS (알림 발송 큐)
      → Lambda (알림 워커)
        ├─ 카카오 알림톡 API (1차: 짧은 알림 + 링크)
        └─ AWS SES (2차: 상세 질문 + 힌트)
```

사용자 설정에 따라 알림 채널 선택 가능(알림톡만, 이메일만, 둘 다). 알림톡 발송 실패 시 자동 SES 폴백.

### 9.2 ECS Fargate 중심 아키텍처 (EKS 미사용)

EKS를 사용하지 않는 결정에 따라, **ECS Fargate**가 컨테이너 오케스트레이션의 중심이 된다.

**ECS Fargate의 장점**:
- 서버리스 컨테이너: EC2 인스턴스 관리 불필요
- Kubernetes 대비 운영 복잡도 대폭 감소
- Auto Scaling 정책으로 트래픽 기반 자동 확장
- ALB(Application Load Balancer)와 네이티브 통합
- 소규모 팀(1~3명)에 적합한 운영 부담

**서비스 구성**:
- `api-service`: FastAPI 메인 API (2~4 태스크, CPU 256/메모리 512부터 시작)
- `worker-service` (V2): 영상 분석 워커 (GPU 필요 시 EC2 Launch Type 혼용)

**Lambda와의 역할 분담**: API 서버처럼 상시 실행이 필요한 서비스는 ECS Fargate, 이벤트 기반 단발 실행(알림 발송, 질문 생성, S3 이벤트 처리)은 Lambda로 분리한다.

### 9.3 월간 인프라 비용 예상

| 규모 | ECS Fargate | RDS | ElastiCache | Lambda/SQS | SES | 합계 |
|------|------------|-----|-------------|------------|-----|------|
| 100명 | ~$15 | ~$15 (db.t3.micro) | ~$13 | 무료 티어 | 무료 | **~$43** |
| 1,000명 | ~$30 | ~$30 (db.t3.small) | ~$26 | ~$5 | ~$3 | **~$94** |
| 10,000명 | ~$80 | ~$100 (db.t3.medium) | ~$52 | ~$20 | ~$30 | **~$282** |

※ LLM API 비용 별도. V2 영상 처리 추가 시 $200~400/월 추가(10,000명 기준).

## 10. 인증(OAuth): 확장 가능한 AWS Cognito 전략

### 10.1 왜 AWS Cognito인가

AWS Cognito는 월 1억 건 이상의 인증을 처리하는 완전 관리형 CIAM(Customer Identity and Access Management) 서비스다. 소셜 로그인, OIDC/SAML 페더레이션, 패스키/패스워드리스 인증을 기본 지원하며, AWS 서비스(Lambda, API Gateway, S3 등)와 네이티브 통합된다. 무료 티어로 MAU 50,000명까지 비용이 발생하지 않는다.

### 10.2 확장 가능한 소셜 로그인 아키텍처

Cognito는 기본적으로 Google, Facebook, Apple, Amazon 소셜 로그인을 직접 지원한다. **카카오, 네이버** 같은 한국 OAuth 프로바이더는 Cognito의 **OIDC(OpenID Connect) 커스텀 프로바이더** 기능을 통해 연동한다.

```
사용자 → Cognito Managed Login → 소셜 IdP 선택
  ├─ Google (네이티브 지원)
  ├─ Apple (네이티브 지원)  
  ├─ 카카오 (OIDC 커스텀 프로바이더)
  └─ 네이버 (OIDC 커스텀 프로바이더)
        → OAuth 인증 → Cognito User Pool에 사용자 매핑
          → JWT 토큰 발급 → API 호출
```

**카카오 로그인 연동 방식**: 카카오는 OIDC를 지원하므로, Cognito User Pool에 OIDC Identity Provider로 등록할 수 있다. Issuer URL, Client ID, Client Secret, 스코프(openid, profile, account_email) 설정으로 연동이 완료된다.

**네이버 로그인**: 네이버는 표준 OIDC를 완전히 지원하지 않을 수 있으므로, Lambda Trigger(Pre Sign-up, Pre Token Generation)를 활용한 커스텀 인증 플로우가 필요할 수 있다.

### 10.3 확장성 고려 사항

| 시나리오 | 대응 방식 |
|---------|----------|
| 새로운 소셜 프로바이더 추가 | Cognito OIDC/SAML 프로바이더 추가 (코드 변경 최소) |
| B2B 기업 SSO | SAML 2.0 페더레이션으로 기업 IdP 연동 |
| 패스워드리스 인증 | Cognito 네이티브 WebAuthn/패스키, SMS/이메일 OTP |
| 멀티 테넌시 (B2B) | Cognito 테넌트 기반 격리 옵션 활용 |
| 글로벌 확장 | Cognito는 다중 리전 배포 지원 |

**핵심**: Cognito의 OIDC 커스텀 프로바이더 기능으로 한국 소셜 로그인을 처리하되, 향후 B2B SAML SSO, 패스워드리스 등으로의 확장을 프로바이더 추가만으로 대응할 수 있는 구조를 처음부터 설계한다.

## 11. 백엔드 프레임워크: FastAPI

**FastAPI** (Python 3.11+)가 본 프로젝트의 백엔드 프레임워크로 최적인 이유:

- **비동기 우선(Async-first)**: LLM API 호출, 외부 서비스 연동에서 비동기 처리가 핵심
- **Pydantic 네이티브 통합**: Instructor와 동일한 Pydantic 모델을 API 스키마와 LLM 출력 스키마에 공유 가능
- **AI 애플리케이션 시장 지배적**: 새로운 Python AI API의 42%가 FastAPI 사용
- **자동 OpenAPI 문서화**: Swagger UI 자동 생성
- **타입 안전성**: 전체 IDE 지원, 자동 완성
- **ECS Fargate 배포 용이**: Docker 컨테이너로 패키징하여 ECR → ECS Fargate 직배포

## 12. 한국 규제 준수: 개인정보보호법(PIPA)

### 12.1 필수 준수 사항

- **이력서 데이터 암호화**: AWS KMS를 사용한 저장 시 암호화(at-rest encryption)
- **데이터 저장 위치**: AWS 서울 리전(ap-northeast-2) 내 저장
- **명시적 동의**: 이용약관과 별도의 한국어 개인정보 수집·이용 동의서
- **데이터 이동성**: 개인정보 열람·이동·삭제 API 구현
- **V2 영상 데이터**: 별도 생체정보 동의 플로우, 녹화 영상 90일 후 자동 삭제
- **최소 수집 원칙**: 서비스에 필요한 최소한의 개인정보만 수집
